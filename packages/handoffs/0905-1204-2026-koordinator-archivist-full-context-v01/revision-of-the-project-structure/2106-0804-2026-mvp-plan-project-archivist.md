MVP-план системы «Архивариус проекта Благополучие»

Дата: 2026-04-08
Статус: рабочий проектный документ
Контур: пакет текущего чата, с последующим выделением канонических документов
Назначение: определить минимально жизнеспособную версию Архивариуса проекта Благополучие и зафиксировать порядок её практической реализации

1. Что такое MVP в рамках этой задачи

MVP означает минимально жизнеспособную версию системы.

В рамках Архивариуса это не “красивая законченная библиотечная платформа”, а первая рабочая сборка, которая уже позволяет получать практическую пользу без попытки сразу реализовать весь замысел.

Главная идея MVP:

сначала сделать маленькую, устойчивую и реально работающую систему поиска и индексации, а уже потом наращивать:

GUI;

расширенные связи;

ревизионные режимы;

медиаконтурные публикации;

сложную классификацию;

красивую навигацию.


2. Главная цель MVP

Главная цель MVP состоит в том, чтобы пользователь мог локально, быстро и без ручного перебора файлов:

найти файл по части имени;

найти файл по фрагменту текста;

увидеть полный путь к найденному файлу;

понять, относится ли файл к пакету чата;

понять, в каком контуре проекта находится файл;

использовать поиск на своём текущем оборудовании без тяжёлой и хрупкой инфраструктуры.


Короткая формула цели:

быстрый локальный поиск по файловому хранилищу проекта с базовым пониманием контекста.

3. Что MVP должен уметь обязательно

Первая рабочая версия должна уметь:

1. Сканировать указанные каталоги внутри Documents/Blagopoluchie.


2. Собирать метаданные файлов:

путь;

имя;

расширение;

размер;

дата изменения.



3. Для поддерживаемых текстовых файлов извлекать текст.


4. Сохранять индекс в SQLite.


5. Выполнять поиск:

по имени;

по части имени;

по пути;

по содержимому.



6. Показывать карточку найденного файла.


7. Определять, находится ли файл внутри пакета чата из 00_INBOX.


8. Обнаруживать служебные файлы:

README-chat-folder.md

.wb-copy-map.tsv

chat-card.md, если он есть.



9. Работать локально из CLI.


10. Не ломать файловую систему и не выполнять опасных действий без явной команды пользователя.



4. Что MVP не должен пытаться делать

Чтобы не утонуть в амбициях раньше первого запуска, MVP не должен на старте пытаться:

строить красивый GUI;

автоматически понимать все темы идеально;

анализировать PDF и изображения;

автоматически удалять или переносить файлы;

делать полноценную многослойную онтологию проекта;

индексировать Documents/repos без отдельного режима;

делать “умную” магическую классификацию без достаточных метаданных;

тянуть внешние сервисы и облачные интеграции.


Короткое правило:

MVP должен быть полезным, а не впечатляющим.

5. Практическая польза первой версии

Даже первая версия уже должна закрывать следующие боли:

пользователь не помнит, где лежит файл;

пользователь помнит кусок имени, но не путь;

пользователь помнит фрагмент текста, но не название файла;

пользователь не хочет глазами перебирать дерево каталогов;

пользователь хочет быстро найти материалы текущих и старых чатовых пакетов;

пользователь хочет увидеть, в каком контуре проекта лежит найденный материал.


6. Минимальный состав модулей MVP

Для первой версии достаточно следующих модулей:

6.1. Модуль конфигурации

Отвечает за:

список индексируемых корней;

исключения;

параметры базы;

режимы запуска.


6.2. Модуль обхода файловой системы

Отвечает за:

рекурсивный обход каталогов;

фильтрацию файлов;

сбор базовых метаданных.


6.3. Модуль извлечения текста

Отвечает за:

чтение поддерживаемых текстовых файлов;

нормализацию текстового содержимого;

получение краткого фрагмента для поиска.


6.4. Модуль базы данных

Отвечает за:

создание SQLite-схемы;

сохранение метаданных;

обновление индекса;

выборки для поиска.


6.5. Модуль поиска

Отвечает за:

поиск по имени;

поиск по пути;

поиск по содержимому;

выдачу результатов в удобной форме.


6.6. Модуль распознавания пакетов чатов

Отвечает за:

определение, что каталог является пакетом чата;

связывание файлов с пакетом;

обнаружение README-chat-folder.md, .wb-copy-map.tsv и chat-card.md.


6.7. CLI-интерфейс

Отвечает за:

команды запуска индексации;

команды поиска;

просмотр карточки файла;

диагностику базы.


7. Предлагаемый минимальный стек

Для MVP разумно использовать:

Python 3

SQLite

стандартную библиотеку Python, где это возможно

argparse для CLI

pathlib для работы с путями

sqlite3 для базы

hashlib для вычисления хэшей

os или pathlib для обхода дерева

при необходимости logging для журнала работы


Главный принцип старта:

минимум зависимостей, максимум предсказуемости.

8. Индексируемые каталоги первой очереди

На старте индексировать:

Documents/Blagopoluchie/00_INBOX

Documents/Blagopoluchie/01_CORE

Documents/Blagopoluchie/02_PROJECTS

Documents/Blagopoluchie/03_LAB

Documents/Blagopoluchie/04_DOCS

Documents/Blagopoluchie/07_ARCHIVE

Documents/Blagopoluchie/09_ADMIN


На старте не индексировать автоматически:

Documents/repos


Если потребуется, этот контур должен подключаться отдельным флагом и отдельной логикой.

9. Форматы файлов первой очереди

На старте поддерживать:

.md

.txt

.tsv

.csv

.json

.yaml

.yml

.log

.sh

.html


Все остальные форматы на первом этапе можно учитывать как файлы без полнотекстового индекса.

10. Минимальная схема данных MVP

Для первой версии допустимо использовать упрощённую схему.

Таблица files

Поля:

id

full_path

relative_path

name

extension

size_bytes

indexed_at

full_text

text_available

project_contour

Поля:

id

Поля:

Это не идеальная модель, но для первого запуска её достаточно.

11. Минимальные команды CLI

Для первой версии полезны такие команды:

init-db

Создаёт пустую базу.

index

Полная индексация заданных корней.

reindex

Переиндексация с обновлением существующих записей.

search-name "строка"

Поиск по имени файла.

search-text "строка"

Поиск по содержимому.

show-file <id>

Показ карточки файла.

show-chat-package <id>

Показ карточки пакета чата.

stats

Общая статистика по базе:

сколько файлов;

сколько текстовых;

сколько пакетов чатов;

дата последней индексации.


12. Минимальная логика поиска

12.1. Поиск по имени

Должен находить:

точное совпадение;

частичное совпадение;

совпадение без учёта регистра.


12.2. Поиск по тексту

Должен искать по full_text среди поддерживаемых форматов.

12.3. Выдача результата

Каждый результат должен содержать:

id;

имя файла;

полный путь;

контур;

признак принадлежности к пакету чата;

короткий фрагмент текста, если поиск был по содержимому.


13. Минимальная логика определения пакета чата

Каталог можно считать пакетом чата, если выполняется хотя бы одно из условий:

1. Он расположен внутри 00_INBOX и содержит README-chat-folder.md.


2. Он расположен внутри 00_INBOX и содержит .wb-copy-map.tsv.


3. Он расположен внутри 00_INBOX и содержит набор файлов, характерный для рабочего пакета чата.



На первом этапе достаточно простой эвристики: если внутри каталога есть README-chat-folder.md или .wb-copy-map.tsv, каталог считается пакетом чата.

14. Что должен показывать файл-карточка в MVP

Карточка файла должна содержать:

id;

имя;

полный путь;

относительный путь;

размер;

дата изменения;

дата индексации;

расширение;

контур проекта;

связан ли файл с пакетом чата;

путь к пакету чата, если связан;

первые строки текста или фрагмент;

служебные признаки:

это README пакета;

это copy-map;

это chat-card.



15. Что должен показывать пакет чата в MVP

Карточка пакета чата должна содержать:

id;

путь пакета;

имя пакета;

путь к README;

путь к copy-map;

путь к chat-card;

количество файлов внутри;

дата последней индексации.


На первом этапе не нужно пытаться идеально различать входные и выходные файлы. Достаточно уже того, что пакет виден как отдельная сущность.

16. Последовательность практической реализации

Этап 1. Каркас проекта

Сделать:

структуру каталогов кода;

базовый README;

конфиг;

скелет CLI.


Результат: есть запускаемый каркас без реальной индексации.

Этап 2. База данных

Сделать:

схему SQLite;

инициализацию базы;

проверку подключения;

базовые операции записи.


Результат: можно создать рабочую базу.

Этап 3. Индексация файловой системы

Сделать:

обход корней;

фильтрацию файлов;

сбор метаданных;

запись метаданных в таблицу files.


Результат: система знает, какие файлы существуют.

Этап 4. Извлечение текста

Сделать:

чтение поддерживаемых форматов;

сохранение full_text и text_excerpt;

мягкую обработку ошибок чтения.


Результат: можно искать по содержимому.

Этап 5. Распознавание пакетов чатов

Сделать:

обнаружение каталогов пакетов;

заполнение таблицы chat_packages;

привязку файлов к пакетам.


Результат: система различает просто файлы и файлы из чатовых пакетов.

Этап 6. CLI-поиск

Сделать:

поиск по имени;

поиск по содержимому;

показ карточки файла;

базовую статистику.


Результат: появляется первый реально полезный пользовательский интерфейс.

Этап 7. Проверка на реальном хранилище

Сделать:

тестовый прогон по реальным каталогам;

исправление ошибок путей и кодировок;

оценку скорости;

чистку грубых багов.


Результат: рабочий MVP, пригодный к использованию.

17. Критерии готовности MVP

MVP считается готовым, если выполняются все условия:

1. База инициализируется одной командой.


2. Индексация проходит по заданным корням без падения на большинстве обычных файлов.


3. Поиск по имени реально находит нужные файлы.


4. Поиск по тексту реально находит нужные markdown и текстовые документы.


5. Поиск показывает полный путь и базовый контекст.


6. Файлы из 00_INBOX связываются с пакетами чатов хотя бы на базовом уровне.


7. Система работает локально на целевом окружении пользователя.



18. Что проверять вручную после первого запуска

После первого запуска нужно проверить:

корректно ли читаются русские имена файлов;

корректно ли читаются длинные пути;

не ломается ли индексатор на битых или странных файлах;

находит ли поиск ожидаемые markdown-документы;

правильно ли определяются пакеты чатов;

не попали ли в индекс лишние каталоги;

не слишком ли тяжёлой получилась база.


19. Что пойдёт сразу после MVP

После запуска первой версии следующим приоритетом разумно сделать:

1. улучшение схемы данных;


2. ручные статусы и темы;


3. поддержку .wb-copy-map.tsv;


4. связи между чатовым пакетом и каноническими копиями;


5. web GUI на localhost.



Это уже будет вторая очередь, а не часть MVP.

20. Основные риски MVP

1. Попытка сразу сделать слишком сложную схему базы.


2. Желание тащить PDF и OCR раньше времени.


3. Попытка сразу прикрутить GUI.


4. Плохая обработка кодировок и спецсимволов в путях.


5. Перегрузка базы полными текстами без ограничений.


6. Слабая диагностика ошибок индексации.



21. Основной принцип разработки MVP

Главный принцип такой:

сначала Архивариус должен научиться честно отвечать на вопрос
“где лежит нужный файл и по какому тексту его можно найти?”

И только потом можно требовать от него:

понимать судьбу документа;

различать каноничность;

вести ревизию;

радовать глаз GUI.


22. Итог

Настоящий документ фиксирует минимально жизнеспособный план реализации Архивариуса проекта Благополучие.

Первая версия должна быть:

локальной;

лёгкой;

понятной;

устойчивой;

полезной уже без графического интерфейса.


Следующим логическим шагом после этого документа является подготовка чернового документа с практической структурой проекта кода Архивариуса:

дерево каталогов репозитория;

состав модулей;

имена файлов;

очередь написания.
