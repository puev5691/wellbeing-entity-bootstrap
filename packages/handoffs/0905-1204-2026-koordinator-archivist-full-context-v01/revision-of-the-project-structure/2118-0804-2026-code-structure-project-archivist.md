Практическая структура проекта кода системы «Архивариус проекта Благополучие»

Дата: 2026-04-08
Статус: рабочий проектный документ
Контур: пакет текущего чата, с последующим выделением канонических документов
Назначение: зафиксировать минимальную и практичную структуру репозитория, модулей и файлов для первой рабочей реализации Архивариуса проекта Благополучие

1. Назначение документа

Этот документ нужен для того, чтобы после ТЗ, модели сущностей и MVP-плана не начинать реализацию с хаоса.

Если предыдущие документы отвечали на вопросы:

зачем нужен Архивариус;

с какими сущностями он работает;

что должно войти в первую рабочую версию;


то этот документ отвечает на вопрос: как именно разложить код, чтобы можно было спокойно писать систему по частям и не плодить кашу из файлов.

Главная задача: дать простую, устойчивую и расширяемую структуру проекта, пригодную для:

локальной разработки;

запуска в Termux и Linux;

постепенного наращивания функциональности;

перехода от CLI к web GUI.


2. Основной принцип структуры кода

На старте Архивариус должен строиться не как “великий универсальный комбайн”, а как набор простых модулей с понятной ответственностью.

Короткая формула:

один модуль = одна ясная роль;

минимум магии;

минимум лишних зависимостей;

простая отладка;

простая сборка в рабочую систему.


Сначала код должен быть удобен для:

запуска;

проверки;

исправления ошибок;

понимания человеком.


И только потом для красоты и архитектурных понтов.

3. Предлагаемый формат репозитория

Для старта разумно сделать один отдельный репозиторий под Архивариуса.

Рабочее имя репозитория допустимо взять таким:

wellbeing-archivist

или

wb-archivist

На текущем этапе предпочтительнее более читаемое имя:

wellbeing-archivist

5. Зачем нужен каждый верхнеуровневой элемент

5.1. README.md

Главный входной файл репозитория.

Должен содержать:

что это за проект;

как установить;

как настроить;

как запустить;

какие команды есть в CLI;

какие каталоги индексируются;

где лежит база.


5.2. pyproject.toml

Современная базовая конфигурация Python-проекта.

Даже если проект останется очень простым, лучше сразу положить его, чем потом на коленке чинить упаковку и запуск.

5.3. requirements.txt

Явный список зависимостей.

На старте он может быть очень коротким или почти пустым.

5.4. .gitignore

Чтобы в Git не летели:

базы SQLite;

временные файлы;

журналы;

кэши;

виртуальные окружения.


5.5. archivist.py

Простой точечный вход в приложение.

Нужен для удобства запуска вида: python archivist.py ...

Можно позже упростить или убрать, но на старте это удобно.

5.6. config/

Каталог конфигурации.

Нужен для:

дефолтных путей;

примеров конфигов;

настройки индексируемых корней;

исключений.


5.7. docs/

Внутренняя инженерная документация по самой системе Архивариус.

Не путать с проектной документацией в Documents/Blagopoluchie. Это документация именно репозитория кода.

5.8. data/

Локальные служебные данные для разработки и примеров.

На старте не использовать для боевой базы пользователя, если это может создать путаницу.

5.9. logs/

Каталог журналов запуска.

5.10. scripts/

Небольшие вспомогательные shell-скрипты.

Полезно для:

быстрой инициализации;

запуска индексации;

повторяемых операций.


5.11. src/archivist/

Основной код системы.

5.12. tests/

Тесты.

Даже если их будет мало, лучше выделить место сразу, чем потом жить в мире “проверил один раз, вроде работало”.

6. Назначение модулей в src/archivist/

6.1. __init__.py

Обозначает пакет и может хранить:

версию;

краткие экспортируемые элементы.


6.2. cli.py

Главный модуль CLI.

Здесь собираются:

аргументы командной строки;

команды;

маршрутизация действий по подкомандам.


Именно он должен понимать команды вроде:

init-db

index

search-name

search-text

show-file

stats


6.3. config.py

Загрузка и валидация конфигурации.

Отвечает за:

чтение JSON-конфига;

пути к индексируемым корням;

путь к базе;

список исключений;

поддерживаемые расширения.


6.4. constants.py

Константы проекта.

Например:

список дефолтных расширений;

имена служебных файлов пакета чата;

значения статусов;

дефолтные пути.


6.5. logging_setup.py

Настройка логирования.

Чтобы не распылять конфигурацию логов по всему проекту.

6.6. models.py

Простые структуры данных уровня приложения.

На старте это могут быть:

dataclass-модели;

типизированные структуры для результата сканирования;

карточки файла;

карточки пакета чата.


Не обязательно сразу городить ORM. Не надо страдать лишним героизмом.

6.7. utils.py

Вспомогательные функции общего назначения.

Но важно: не превращать этот модуль в кладбище всего подряд.

6.8. scanner.py

Модуль обхода файловой системы.

Отвечает за:

рекурсивный обход;

фильтрацию;

определение типов файлов;

сбор базовых метаданных;

безопасную работу с путями.


6.9. text_extract.py

Модуль извлечения текста.

Отвечает за:

чтение текстовых файлов;

нормализацию текста;

ограничение объёма;

получение text_excerpt;

обработку ошибок кодировки.


6.10. db.py

Модуль базы данных SQLite.

Отвечает за:

создание схемы;

подключение;

транзакции;

запись файлов;

запись пакетов чатов;

базовые выборки.


6.11. indexer.py

Оркестратор индексации.

Это один из главных модулей MVP.

Он связывает:

конфиг;

scanner;

text_extract;

db;

chat_packages.


Именно он реализует:

полную индексацию;

переиндексацию;

обновление записей.


6.12. search.py

Модуль поиска.

Отвечает за:

поиск по имени;

поиск по пути;

поиск по содержимому;

поиск по пакетам чатов;

возврат структурированных результатов.


6.13. chat_packages.py

Модуль логики пакетов чатов.

Отвечает за:

определение пакета чата;

обнаружение README-chat-folder.md;

обнаружение .wb-copy-map.tsv;

обнаружение chat-card.md;

связывание файлов с пакетом.


На старте это очень важный модуль, потому что именно он делает Архивариуса не просто файловым поиском, а инструментом работы с твоей реальной системой.

6.14. presenters.py

Модуль представления результатов.

Отвечает за:

человекочитаемый вывод карточек;

форматирование списков;

подготовку CLI-вывода.


Это позволяет не мешать бизнес-логику и оформление.

6.15. stats.py

Модуль сводной статистики.

Отвечает за:

общее число файлов;

число текстовых файлов;

число пакетов чатов;

число файлов по контурам;

время последней индексации.


7. Что должно быть в конфигурации

На старте конфиг можно сделать в JSON.

Минимальные параметры:

путь к SQLite-базе;

список индексируемых корней;

список исключаемых каталогов;

список поддерживаемых расширений;

максимальный размер файла для полнотекстового чтения;

режим пересоздания индекса;

режим логирования.


Пример логики полей:

database_path

roots

exclude_dirs

text_extensions

max_text_file_size_bytes

log_level


8. Минимальная структура команды запуска

На старте удобно поддержать такой стиль:

python archivist.py <command> [options]

Примеры:

python archivist.py init-db

python archivist.py index

python archivist.py reindex

python archivist.py search-name "archivist"

python archivist.py search-text "управляющий контур"

python archivist.py show-file 15

python archivist.py stats


Это простой и неубиваемый формат для старта.

9. Очередность написания файлов кода

Чтобы не метаться по репозиторию, писать лучше в таком порядке.

Шаг 1. Базовый каркас

Создать:

README.md

.gitignore

pyproject.toml

requirements.txt

archivist.py

src/archivist/__init__.py

src/archivist/cli.py

src/archivist/config.py


Результат: проект уже существует как запускаемая заготовка.

Шаг 2. Константы и модели

Создать:

constants.py

models.py


Результат: появляется единый язык для внутренних данных.

Шаг 3. Работа с базой

Создать:

db.py


Сделать:

инициализацию схемы;

создание таблиц;

базовые вставки и выборки.


Результат: есть живая база.

Шаг 4. Сканирование файлов

Создать:

scanner.py


Сделать:

обход корней;

фильтрацию;

сбор метаданных.


Результат: система видит файловую систему.

Шаг 5. Извлечение текста

Создать:

text_extract.py


Сделать:

чтение поддерживаемых форматов;

нормализацию;

защиту от ошибок.


Результат: можно индексировать содержимое.

Шаг 6. Индексатор

Создать:

indexer.py


Сделать:

сбор информации из scanner;

извлечение текста;

запись в базу.


Результат: появляется первый реальный проход индексации.

Шаг 7. Пакеты чатов

Создать:

chat_packages.py


Сделать:

определение пакетов;

связь файлов с пакетами;

запись в таблицу chat_packages.


Результат: Архивариус начинает понимать твою систему работы.

Шаг 8. Поиск и вывод

Создать:

search.py

presenters.py

stats.py


Результат: появляется рабочий CLI-интерфейс.

Шаг 9. Логи и удобство запуска

Создать:

logging_setup.py

скрипты из scripts/


Результат: систему проще отлаживать и запускать регулярно.

10. Минимальная схема ответственности между файлами

Чтобы код не начал течь во все стороны, надо держать такую дисциплину:

cli.py не лезет глубоко в SQL;

db.py не форматирует красивый вывод;

scanner.py не знает про команды CLI;

text_extract.py не принимает решений о каноничности;

search.py не ходит напрямую по файловой системе;

presenters.py ничего не решает, а только показывает.


Это скучно, зато потом не приходится выковыривать логику из помойки.

На старте не обязательно делать:

api.py

topic_classifier.py

Их можно добавить позже, когда MVP уже станет полезным.

12. Когда имеет смысл выделять новые модули

Новый модуль стоит выделять тогда, когда одновременно выполняются условия:

1. В коде появляется самостоятельная ответственность.


2. Логика становится больше нескольких коротких функций.


3. Её хочется тестировать отдельно.


4. Её неприятно тащить в уже существующий файл без засорения.



Если этих признаков нет, дробить код ради красоты не надо.

13. Минимальный набор тестов первой очереди

В первой волне тестов достаточно проверить:

test_config.py

читается ли конфиг;

корректно ли валидируются пути и списки.


test_scanner.py

находит ли scanner файлы;

пропускает ли исключённые каталоги;

определяет ли расширения.


test_text_extract.py

читает ли utf-8;

не падает ли на мусорных кодировках;

корректно ли режет фрагменты.


test_db.py

создаётся ли схема;

пишутся ли записи;

читаются ли записи обратно.


test_indexer.py

проходит ли полный цикл индексирования.


test_search.py

находит ли по имени;

находит ли по тексту.


test_chat_packages.py

распознаёт ли каталог пакета чата;

видит ли README;

видит ли copy-map;

связывает ли файлы с пакетом.

1. Что такое Архивариус.


2. Что он умеет в MVP.


3. Как установить зависимости.


4. Как инициализировать базу.


5. Как запустить индексацию.


6. Как искать по имени.


7. Как искать по тексту.

Если этого не будет, через неделю даже ты сам будешь смотреть на репозиторий с выражением “кто это сделал и зачем”.

15. Практическая стратегия разработки

Для Архивариуса наиболее разумна такая стратегия:

сначала добиться запуска CLI;

потом добиться индексации;

потом добиться поиска;

потом добиться распознавания пакетов чатов;

потом чистить структуру;

потом добавлять GUI.


Не наоборот.

16. Минимальный план коммитов

Чтобы работа не превратилась в один гигантский нечитаемый коммит, можно идти такими блоками:

1. init project skeleton


2. add sqlite schema and db init


3. add filesystem scanner


4. add text extraction for basic formats


5. add indexer flow


6. add basic search commands


7. add chat package detection


8. add stats and formatted output


9. add tests for mvp modules



Да, коммиты тоже стоит продумывать. Мир и так полон бессмысленного шума.

17. Что будет следующим документом после этого

После фиксации структуры проекта кода логично собрать один из двух документов:

либо черновую спецификацию SQLite-схемы;

либо стартовый план реализации по дням или шагам.


Для практического старта полезнее сначала сделать именно: черновую спецификацию SQLite-схемы MVP.

18. Итог

Настоящий документ фиксирует практическую структуру репозитория и модулей системы Архивариус проекта Благополучие.

Эта структура должна помочь:

не захламить код с самого старта;

писать систему по модулям;

отлаживать её без боли;

постепенно перейти от простого CLI-индексатора к полноценному локальному архивариусу проекта.
