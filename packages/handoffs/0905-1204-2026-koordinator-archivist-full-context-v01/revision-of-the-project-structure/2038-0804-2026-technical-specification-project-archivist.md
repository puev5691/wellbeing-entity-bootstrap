Техническое задание на систему «Архивариус проекта Благополучие»

Дата: 2026-04-08
Статус: рабочее ТЗ
Контур: пакет текущего чата, с последующим выделением канонических документов
Назначение: определить требования к локальной системе индексирования, поиска, навигации и учёта файловых и чатовых материалов проекта Благополучие

1. Основание для постановки задачи

В проекте Благополучие накоплено значительное количество:

чатов;

рабочих пакетов чатов;

handoff-файлов;

регламентов;

черновиков;

тематических документов;

технических заметок;

журналов;

промежуточных и архивных материалов.


Существующая проблема заключается не только в росте числа файлов, но и в усложнении связи между:

темами;

чатами;

входными материалами;

итоговыми файлами;

каноническими каталогами;

архивными слоями;

актуальными и устаревшими версиями.


У пользователя отсутствует удобный и быстрый механизм, позволяющий:

мгновенно находить нужный файл;

понимать, из какого чата он появился;

видеть, где находится каноническая версия;

различать актуальные, архивные и промежуточные материалы;

находить данные не только по имени файла, но и по теме, контексту и содержимому.


В связи с этим требуется спроектировать систему «Архивариус проекта Благополучие».

2. Рабочее название системы

Рабочее название: Архивариус проекта Благополучие

Допустимые смысловые обозначения:

архивариус;

поисковик проекта;

библиотечный демон порядка;

локальный индексатор и навигатор проектного контента.


Во всех проектных документах на текущем этапе базовым названием считать: Архивариус проекта Благополучие.

3. Назначение системы

Система Архивариус предназначена для:

индексации файлового хранилища проекта;

учёта рабочих пакетов чатов;

быстрого поиска по именам, путям, темам и содержимому;

фиксации связей между чатами, файлами и каноническими каталогами;

помощи в ревизии, сортировке, архивировании и актуализации материалов;

снижения зависимости от памяти пользователя при поиске информации;

поддержки управляемой структуры знаний проекта.


4. Главная цель системы

Главная цель системы состоит в том, чтобы пользователь в любой момент мог быстро найти нужную информацию без необходимости:

держать в голове всё дерево каталогов;

перечитывать длинные списки имён файлов;

вручную вспоминать, в каком чате и в каком каталоге создавался материал;

открывать множество файлов подряд ради проверки содержимого.


Короткая формула цели:

найти нужный материал быстро, понятно, с контекстом происхождения и статусом актуальности.

5. Основные задачи системы

Система должна решать следующие задачи:

1. Сканировать файловое хранилище проекта.


2. Формировать индекс файлов и каталогов.


3. Извлекать текстовое содержимое поддерживаемых форматов.


4. Хранить метаданные и поисковый индекс в локальной базе.


5. Позволять искать по:

имени файла;

части имени;

пути;

теме;

тегам;

статусу;

содержимому;

связи с конкретным чатом.



6. Показывать происхождение файла:

из какого чата он появился;

входной он или выходной;

куда был разложен по каноническим каталогам.



7. Показывать статус файла:

актуален;

актуален, но требует переработки;

черновик;

архив;

дубликат;

кандидат на удаление.



8. Поддерживать ревизию хранилища и принятие решений о перемещении, архивировании и актуализации.


9. Обеспечивать основу для дальнейшего GUI и тематической навигации.



6. Границы системы

6.1. Что входит в систему

На текущем этапе в систему входят:

файловое хранилище внутри Documents/Blagopoluchie;

рабочие пакеты чатов, размещённые в 00_INBOX;

канонические каталоги проекта;

архивные каталоги;

файлы метаданных чатов;

карты раскладки файлов;

текстовые и markdown-документы проекта;

локальная база индекса.


6.2. Что не входит в систему автоматически

На текущем этапе в систему не входит автоматически:

внутреннее содержимое чатов ChatGPT, если оно не выгружено в файлы;

внешние облачные сервисы без отдельной интеграции;

Git-репозитории в Documents/repos, если для них не будет отдельного режима обработки;

медиафайлы, содержимое которых невозможно надёжно индексировать без специальных модулей;

автоматическое принятие решений об удалении файлов без участия пользователя.


6.3. Принцип ограничения

Архивариус не должен делать вид, будто он знает всё. Он должен честно работать только с тем, что реально доступно в файловой системе и описано в метаданных.

7. Основные сущности системы

На текущем этапе нужно различать следующие сущности.

7.1. Файл

Единичный объект файловой системы, имеющий:

путь;

имя;

расширение;

размер;

дату изменения;

хэш;

текстовое содержимое, если оно извлекается;

статус;

тему;

принадлежность к чату или каноническому контуру.


7.2. Каталог

Структурная единица хранения, включающая файлы и подкаталоги.

7.3. Пакет чата

Каталог, связанный с одним конкретным рабочим чатом и содержащий:

входные материалы;

файлы контекста;

результаты работы чата;

README чата;

карту раскладки;

при необходимости карточку чата.


7.4. Канонический файл

Файл, признанный основной версией материала по теме и размещённый в целевом смысловом контуре проекта.

7.5. Архивный файл

Файл, сохранённый как исторически значимый, но не являющийся актуальной рабочей версией.

7.6. Тема

Смысловое направление, по которому могут быть сгруппированы чаты и файлы.

7.7. Статус

Признак текущего состояния файла или пакета:

актуален;

черновик;

архив;

требует переработки;

дубликат;

кандидат на удаление;

не классифицирован.


7.8. Карта раскладки

Файл .wb-copy-map.tsv, определяющий соответствие между файлами рабочего пакета чата и каноническими каталогами проекта.

7.9. Карточка чата

Краткий метаданный файл, описывающий:

тему чата;

назначение;

входные файлы;

ключевые результаты;

связи с другими чатами;

статус;

канонические выходы.


8. Основные пользовательские сценарии

8.1. Поиск по строке

Пользователь вводит слово, часть имени, тему или фрагмент текста и получает список релевантных файлов.

8.2. Поиск по теме

Пользователь выбирает тему и получает:

связанные чаты;

канонические файлы;

черновики;

архив;

последние актуальные материалы.


8.3. Поиск происхождения

Пользователь открывает файл и видит:

из какого чата он появился;

был ли он входным или выходным;

есть ли каноническая копия;

где она лежит;

есть ли связанные версии.


8.4. Поиск актуальной версии

Пользователь хочет не любой файл по теме, а именно актуальный. Система должна помочь отличить:

канонический файл;

черновой промежуточный;

устаревший;

архивный.


8.5. Ревизия каталога

Пользователь открывает каталог и получает возможность:

увидеть список файлов;

отфильтровать по статусу;

отметить дубликаты;

пометить кандидатов на архивирование;

видеть неразобранные материалы.


8.6. Работа с пакетом чата

Пользователь открывает пакет конкретного чата и видит:

входные файлы;

выходные файлы;

файлы handoff;

карту раскладки;

связанные канонические документы;

статус пакета.


9. Функциональные требования к MVP

Минимально жизнеспособная версия системы должна уметь:

1. Сканировать заданные каталоги проекта.


2. Индексировать:

имена файлов;

пути;

размеры;

даты изменения;

расширения;

текстовое содержимое markdown, txt, tsv, md и аналогичных форматов.



3. Сохранять данные в локальной базе SQLite.


4. Выполнять поиск по:

имени;

части имени;

пути;

содержимому.



5. Показывать карточку найденного файла:

полный путь;

размер;

дата изменения;

краткий фрагмент текста;

статус, если он определён;

принадлежность к пакету чата, если она определена.



6. Обнаруживать файлы .wb-copy-map.tsv.


7. На основании карт раскладки строить связь между файлом из пакета чата и каноническим каталогом.


8. Поддерживать ручную метку темы и статуса.


9. Работать локально без обязательного доступа к интернету.


10. Быть пригодной для запуска на имеющемся у пользователя оборудовании, в том числе в Termux и на Linux-средах.



10. Требования ко второй очереди развития

После MVP система должна получить следующие возможности:

1. Web GUI на localhost.


2. Карточки чатов.


3. Тематический индекс.


4. Выявление дубликатов по хэшу и по имени.


5. Отображение исторических и канонических версий.


6. Поддержку фильтров:

тема;

статус;

тип файла;

каталог;

дата;

размер.



7. Отдельный режим для ревизии хранилища.


8. Отдельный режим для работы с архивом.


9. Учёт заметок пользователя по файлам.


10. Экспорт отчётов ревизии.



11. Нефункциональные требования

Система должна быть:

11.1. Локальной

Основные функции поиска и навигации должны работать без внешних серверов.

11.2. Прозрачной

Пользователь должен понимать:

откуда взялись данные;

когда индекс обновлялся;

почему файл относится к той или иной категории;

какие решения приняты автоматически, а какие вручную.


11.3. Осторожной

Система не должна автоматически удалять или безвозвратно перемещать материалы без явного действия пользователя.

11.4. Расширяемой

Архитектура должна позволять добавлять:

новые типы индексации;

новые сущности;

новые фильтры;

GUI;

режимы интеграции с дополнительными контурами.


11.5. Пригодной для слабого и нестабильного окружения

Нужно учитывать:

работу с Android/Termux;

ограниченность ресурсов;

возможные проблемы с длинными путями;

необходимость работы без тяжёлых зависимостей на старте.


12. Предпочтительная архитектура решения

На текущем этапе предпочтительная архитектура следующая:

12.1. Индексатор

Фоновый или запускаемый вручную модуль, обходящий файловое дерево и обновляющий базу.

12.2. Локальная база

SQLite как основное хранилище метаданных и индекса на старте.

12.3. Поисковый слой

Модуль, выполняющий запросы по базе и возвращающий:

список файлов;

совпадения по тексту;

карточки объектов;

связанные сущности.


12.4. API или сервисный слой

Локальный слой, через который GUI или CLI получают данные.

12.5. GUI

Web-интерфейс на localhost как приоритетный вариант пользовательского интерфейса.

13. Предпочтительный стек для старта

На старте допустим и предпочтителен следующий прагматичный подход:

Python для основного прототипа;

SQLite для базы;

стандартные средства обхода каталогов и вычисления хэшей;

извлечение текста из простых форматов без тяжёлых зависимостей;

Flask или аналогичный лёгкий web-фреймворк для GUI второй очереди;

CLI-режим для первых запусков и отладки.


Это не исключает перехода на другой стек позже, если появятся серьёзные основания.

14. Источники данных для индексирования

На первом этапе индексировать:

Documents/Blagopoluchie/00_INBOX

Documents/Blagopoluchie/01_CORE

Documents/Blagopoluchie/02_PROJECTS

Documents/Blagopoluchie/03_LAB

Documents/Blagopoluchie/04_DOCS

Documents/Blagopoluchie/07_ARCHIVE

Documents/Blagopoluchie/09_ADMIN


Documents/repos на старте не индексировать автоматически, либо подключать отдельным флагом, потому что это отдельный технический контур и его нельзя бездумно смешивать с документационным слоем.

.tsv

.yml

.html

изображения с OCR, если это вообще потребуется и будет оправдано.


16. Метаданные, которые система должна хранить

Для файла:

уникальный идентификатор;

полный путь;

относительный путь;

имя;

расширение;

размер;

дата изменения;

дата индексации;

хэш;

тема;

статус;

признак каноничности;

признак архивности;

ссылка на пакет чата;

текстовый фрагмент;

заметка пользователя.


Для пакета чата:

идентификатор;

путь;

имя;

тема;

статус;

дата последней активности;

README;

chat-card;

список входных файлов;

список выходных файлов;

связанные канонические файлы.


17. Правила автоматизации и ручного контроля

Система может автоматически:

индексировать имена и содержимое;

обнаруживать карты раскладки;

связывать файлы по путям и хэшам;

предлагать дубликаты;

предлагать кандидатов на архивирование по правилам.


Система не должна автоматически без подтверждения:

удалять файлы;

назначать окончательный статус;

объявлять файл каноническим;

заменять одну версию другой;

переносить файлы между контурами.


Человеческая ошибка и так вездесуща, не надо давать ей мотор и автопилот.

18. Критерии успеха MVP

MVP можно считать успешным, если пользователь сможет:

1. За несколько секунд найти нужный файл по части имени или фрагменту текста.


2. Понять, где лежит файл и к какому контуру он относится.


3. Увидеть, связан ли файл с пакетом чата.


4. Увидеть, куда файл должен быть разложен по карте раскладки, если такая карта есть.


5. Получить список файлов по теме или каталогу без ручного перебора дерева.


6. Использовать систему локально в своей рабочей среде.



19. Критерии качества будущей полной системы

Полная система должна позволять:

работать с файловым хранилищем как с осмысленной библиотекой, а не как с кладбищем файлов;

быстро находить нужные материалы по теме, а не только по имени;

видеть происхождение и судьбу документов;

различать рабочий пакет, канонический файл, архивный след и дубликат;

поддерживать ревизию и дальнейший рост проекта.


20. Основные риски

1. Перегрузка системы лишними функциями на старте.


2. Попытка сразу сделать красивый GUI без устойчивой модели данных.


3. Смешение чатового, документационного и репозиторного контуров.


4. Слабая дисциплина метаданных в пакетах чатов.


5. Ложная уверенность в автоматической классификации.


6. Падение производительности при слишком широком охвате без фильтрации.


7. Преждевременное подключение тяжёлых форматов вроде PDF и OCR.

24. Итог

Настоящее ТЗ фиксирует постановку задачи на создание первой версии системы Архивариус проекта Благополучие.

На текущем этапе приоритетом является не красота интерфейса, а создание устойчивого ядра:

модели данных;

индексатора;

поиска;

связи с пакетами чатов;

понятной навигации по файловому хранилищу.
