# Канон служебного пространства Архивариуса и идея конфигурации исключений

Дата: 2026-04-09
Статус: рабочий проектный документ
Контур: пакет текущего чата
Назначение: зафиксировать каноническую структуру служебного пространства Архивариуса и идею конфигурации, при которой Архивариус не должен индексировать собственную служебную зону

## 1. Зачем Архивариусу отдельное пространство

По мере взросления Архивариус начинает порождать собственные артефакты:

- базы индексации
- отчёты
- манифесты
- кэш
- снимки
- служебные логи
- промежуточные файлы подготовки

Если эти артефакты держать вперемешку с живым файловым болотом проекта, Архивариус начнёт индексировать собственные выделения и плодить бюрократическое самопоедание.

Следовательно, Архивариусу требуется собственная служебная зона.

## 2. Канонический путь служебной зоны

Рекомендуемый корень:

`/storage/emulated/0/Documents/Blagopoluchie/_ARCHIVIST`

## 3. Каноническая структура

Внутри рабочей зоны Архивариуса рекомендуется держать:

- `db/` — базы индексации
- `configs/` — конфигурации различных прогонов
- `reports/` — общие отчёты
- `manifests/` — манифесты контекста
- `repo_manifests/` — манифесты репозиториев
- `repo_reports/` — отчёты по репозиториям
- `snapshots/` — снимки состояния
- `cache/` — служебный кэш
- `logs/` — логи работы
- `quarantine/` — карантин для спорных или тестовых файлов
- `tmp/` — временные файлы

## 4. Главное правило

Архивариус не должен индексировать собственную служебную зону по умолчанию.

Иначе получится порочный цикл:

- Архивариус пишет отчёт
- индексирует отчёт
- пишет отчёт об индексации отчёта
- снова индексирует это
- и бодро идёт к административному безумию

## 5. Текущее состояние реализации

На текущем этапе текущая версия Архивариуса ещё не реализует полноценную поддержку `exclude_paths` как рабочую функцию.

Следовательно, до реализации такого механизма нужно соблюдать практическое правило:

- либо не наводить Архивариуса на корень, включающий его служебную зону
- либо проектировать прогоны так, чтобы `_ARCHIVIST` находился вне текущего индексируемого контура

## 6. Идея будущей конфигурации

В будущем конфиг Архивариуса должен поддерживать не только:

- корневой каталог индексации
- путь к базе
- режимы логирования

но и:

- `exclude_paths`
- возможно `exclude_name_patterns`
- возможно `exclude_extensions`
- отдельный `workspace_root`

## 7. Практический смысл

Введение служебной зоны Архивариуса и исключений для неё является обязательным условием дальнейшего роста инструмента в сторону:

- больших файловых пространств
- многоконтурной индексации
- репозиторного режима
- подготовки контекста для кодовых Сущностей

## 8. Итог

Служебное пространство Архивариуса следует считать обязательной частью дальнейшей архитектуры.

Поддержку исключения собственной зоны из индексации следует считать одним из ближайших инженерных шагов.
