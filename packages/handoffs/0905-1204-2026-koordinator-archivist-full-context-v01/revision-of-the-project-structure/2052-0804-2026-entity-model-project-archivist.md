Модель сущностей системы «Архивариус проекта Благополучие»

Дата: 2026-04-08
Статус: рабочий проектный документ
Контур: пакет текущего чата, с последующим выделением канонических документов
Назначение: зафиксировать основные сущности, связи и роли данных в системе Архивариус проекта Благополучие

1. Назначение документа

Этот документ уточняет и развивает техническое задание на систему Архивариус проекта Благополучие.

Если ТЗ отвечает на вопрос: что должна делать система,

то данный документ отвечает на вопрос: с какими сущностями она должна работать и как эти сущности связаны между собой.

Главная задача документа: не дать системе превратиться в безликий поисковик по файлам, а сразу заложить в неё способность различать:

файл;

пакет чата;

тему;

каноническую версию;

архивный след;

карту раскладки;

статус;

заметку;

контур проекта.


2. Общий принцип модели

Архивариус должен понимать файловое хранилище проекта не как хаотическую совокупность путей, а как библиотеку взаимосвязанных объектов.

Базовый принцип модели:

есть физические объекты файловой системы;

есть смысловые сущности, которые описывают роль объектов;

есть связи происхождения, принадлежности, актуальности и раскладки;

есть ручные и автоматические признаки;

есть различие между реальным содержимым и интерпретацией этого содержимого.


Иными словами: один и тот же файл может существовать физически как файл, но при этом в модели играть роль:

входного контекста чата;

чернового артефакта;

канонической версии;

архивной копии;

элемента темы;

объекта ревизии.


3. Основные уровни модели

Для ясности модель делится на несколько уровней.

3.1. Физический уровень

То, что реально существует в файловой системе:

каталоги;

файлы;

пути;

даты;

размеры;

содержимое;

хэши.


3.2. Контекстный уровень

То, что описывает происхождение и окружение:

пакет чата;

входной файл;

выходной файл;

карта раскладки;

карточка чата;

контур проекта.


3.3. Смысловой уровень

То, что позволяет ориентироваться в содержании:

тема;

статус;

каноничность;

архивность;

тип материала;

связи между документами.


3.4. Управленческий уровень

То, что нужно для ревизии и поддержания порядка:

решения пользователя;

заметки;

кандидаты на перенос;

кандидаты на архивирование;

дубликаты;

связи с отчётами ревизии.

На текущем этапе модель должна включать следующие сущности.

ChatPackage является ключевой сущностью для связи между содержанием чатов и файловой системой проекта.

Нужна для различения ролей файлов внутри пакета чата. Один файл не просто “лежит в папке”, а выполняет понятную функцию в жизненном цикле конкретного чата.

6. Основные связи между сущностями

Ниже приведены основные связи модели.

6.1. Directory -> File

Один каталог содержит много файлов.

Связь:

один ко многим


6.2. Directory -> Directory

Каталог может иметь дочерние каталоги.

Связь:

рекурсивная один ко многим


6.3. ChatPackage -> ChatArtifact

Один пакет чата содержит много артефактов.

Связь:

один ко многим


6.4. ChatArtifact -> File

Один артефакт связан с одним файлом.

Связь:

многие к одному


6.5. Topic -> File

Одна тема может быть связана со многими файлами.

Связь:

один ко многим на старте

многие ко многим на следующем этапе, если понадобится многотематичность


6.6. Topic -> ChatPackage

Одна тема может объединять несколько пакетов чатов.

Связь:

один ко многим


6.7. Status -> File

Один статус может быть присвоен многим файлам.

6.8. Status -> ChatPackage

Один статус может быть присвоен многим пакетам чатов.

6.9. ChatPackage -> CopyMap

Один пакет чата может иметь много записей карты раскладки.

6.10. CopyMap -> File

Одна запись карты раскладки относится к одному исходному файлу.

6.11. File -> CanonicalRelation -> File

Один файл может быть связан со многими другими файлами через разные типы отношений.

6.12. File -> RevisionDecision

По одному файлу может быть принято много решений в разное время.

6.13. ProjectContour -> Directory

Один контур проекта объединяет много каталогов.

6.14. ProjectContour -> File

Файл принадлежит одному основному контуру по месту хранения, но может иметь смысловые связи с другими контурами.

7. Минимальная логика идентификации объектов

Для устойчивой работы модель должна различать:

физическую идентичность;

смысловую идентичность;

версионную связь.


7.1. Физическая идентичность файла

Определяется через:

полный путь;

размер;

время изменения;

хэш.


7.2. Смысловая идентичность документа

Определяется не только по файлу, но и по тому, является ли он:

вариантом одного и того же текста;

канонической копией;

архивной копией;

переработанной версией;

черновиком по отношению к другому документу.


7.3. Идентичность пакета чата

Определяется по пути каталога и его функции как контейнера одного рабочего чата.

8. Логические типы файлов

Для удобства поиска и фильтрации помимо расширения полезно ввести логические типы файлов.

Рекомендуемые значения:

markdown_document

plain_text

table_tsv

table_csv

json_data

yaml_config

shell_script

html_document

log_file

binary_other


Это позволит пользователю искать не только “.md”, но и “все табличные карты”, “все shell-скрипты” и так далее.

9. Логические роли файлов внутри пакета чата

Необходимо различать роли файлов внутри пакета чата.

Рекомендуемые роли:

входной контекст;

основной handoff;

дополнительный handoff;

рабочий черновик;

финальный результат;

карта раскладки;

README пакета;

карточка чата;

технический вспомогательный файл;

журнал работы.


Это различие особенно важно для поиска по пакету, потому что пользователю чаще нужен не любой файл из папки чата, а конкретный тип файла.

10. Логика каноничности

Система должна различать:

файл, родившийся в чате;

файл, ставший канонической копией;

файл, оставшийся только следом происхождения;

архивную копию;

заменённую версию.


10.1. Признак каноничности

Признак is_canonical сам по себе недостаточен, если не видно, для чего именно файл каноничен.

Поэтому на практике каноничность должна опираться на:

место хранения;

решение ревизии;

связь через CanonicalRelation;

указание в карте раскладки;

при необходимости ручное подтверждение пользователя.


10.2. Канонический файл не равен лучшему автоматически

Канонический файл - это не “самый умный” или “самый длинный”, а тот, который принят как основной рабочий экземпляр в целевом контуре.

11. Логика архива

Архивность тоже нельзя сводить только к расположению в 07_ARCHIVE.

Файл может быть архивным по смыслу, даже если физически ещё не перенесён в архивный контур.

Поэтому полезно различать:

is_archive как смысловой флаг;

физическое нахождение в архивном каталоге;

отношение archive_copy_of.


12. Логика темы

На старте достаточно одной основной темы на файл и одной основной темы на пакет чата.

Однако нужно предусмотреть рост модели:

один файл может относиться к нескольким темам;

один пакет чата может работать на стыке нескольких тем;

одна тема может включать документы из разных контуров.


Поэтому в дальнейшем желательно перейти от поля topic_id к отдельной таблице связей EntityTopicLink.

На этапе MVP допустима упрощённая модель с одной основной темой.

13. Логика статусов

Статус не должен путаться с типом файла, темой или контуром хранения.

Например:

markdown-документ может быть черновиком, архивом или каноническим файлом;

файл в 03_LAB не обязан быть черновиком навсегда;

файл в 07_ARCHIVE может сохранять высокую ценность, несмотря на архивный статус.


Статус отвечает на вопрос: каково текущее рабочее положение объекта.

14. Логика медиаконтурных публикаций

С учётом новой практики проекта нужно предусмотреть, что один файл может быть связан с публикацией во внешнем медиаконтуре.

Базовая цепочка такая:

исходный файл проекта;

публикация на telegra.ph;

фиксация публикации в Telegram-канале;

обсуждение в чате канала;

возможная переработка исходного файла.


Следовательно, в будущей модели желательно поддержать сущность Publication и связи:

published_from_file

discussed_in_channel

derived_revision_after_discussion


Это важно для трассировки судьбы публичных материалов.

15. Минимальная ER-логика MVP

Для первой рабочей версии Архивариуса достаточно следующего минимального ядра:

File

Directory

ChatPackage

ChatArtifact

Topic

Status

ProjectContour

CopyMap

CanonicalRelation

RevisionDecision

UserNote


Это уже позволит:

индексировать файлы;

видеть пакеты чатов;

искать по содержимому и контексту;

хранить статусы;

связывать пакет с каноническими копиями;

фиксировать решения ревизии.


16. Что можно упростить на старте

Чтобы не утонуть в собственной важности раньше запуска MVP, допустимо временно упростить:

1. Topic оставить как одну основную тему.


2. UserNote хранить как поле в File и ChatPackage, а не как отдельную таблицу.


3. RevisionDecision вводить после первого рабочего поиска.


4. Publication оставить на вторую очередь.


5. SearchHit не хранить постоянно, а формировать на лету.


6. Поддержку нескольких типов отношений между версиями сначала сократить до:

canonical_copy_of

archive_copy_of

revised_version_of




Это позволит быстрее получить рабочее ядро.

17. Основные вопросы, которые модель должна помогать решать

Хорошая модель Архивариуса должна позволять ответить на вопросы:

Где находится нужный файл?

К какому контуру он относится?

К какой теме он относится?

Является ли он актуальным?

Есть ли у него каноническая версия?

Из какого чата он появился?

Есть ли карта раскладки для него?

Есть ли архивная копия?

Какие решения по нему уже принимались?

Какие ещё файлы связаны с ним по теме или происхождению?


Если модель не даёт ответить на эти вопросы, значит она декоративная и бесполезная.

18. Итоговая формула модели

Архивариус должен видеть проектное хранилище как систему, где:

физически существуют файлы и каталоги;

контекстно существуют пакеты чатов и карты раскладки;

смыслово существуют темы, статусы и канонические связи;

управленчески существуют решения ревизии и заметки пользователя.


Только сочетание этих четырёх слоёв делает поиск не просто быстрым, а действительно полезным.
